# hudukaata end-to-end test

`run.sh` is a self-contained integration test that exercises the full pipeline
in two phases: an initial index of a batch of media, followed by an incremental
update that adds a new file and verifies the updated index is served correctly.

## What it tests

```
Phase 1 -- initial index
  batch1/ (blue_sky.png + green_field.png)
      |
      v
  indexer  --folder batch1
      |  writes ChromaDB store (2 documents)
      v
  search API  (background, port 18080)
  webapp      (background, port 15173)
      |
  assertions
    1. GET /                       -> response body contains <html
    2. GET /api/search?q=blue+sky  -> >= 1 result via webapp proxy
      |
      v
  services stopped

Phase 2 -- incremental update
  batch2/ (red_sunset.png -- generated by PIL, not committed)
      |
      v
  indexer  --folder batch2   (loads existing DB, adds 1 new file)
      |  updated ChromaDB store (3 documents total)
      v
  search API  (restarted)
  webapp      (restarted)
      |
  assertions
    3. GET /                       -> response body contains <html
    4. GET /api/search?q=blue+sky  -> >= 3 results
                                   -> at least one result path contains "batch2"
                                      (proves the incremental update was applied)
```

The proxy approach (steps 2 and 4) is deliberate: rather than calling the
search API directly, the test hits the webapp's `/api` route.  This verifies
that the webapp is running *and* is correctly wired to the search API.

## Why the result counts

- **Phase 1** asserts `>= 1` — existence check; both sample images may be
  returned for a "blue sky" query but only one is required.
- **Phase 2** asserts `>= 3` — all three documents (batch1 x2, batch2 x1) must
  be in the index.  The store returns the top-N most similar items with no
  minimum relevance threshold, so with `n=5` all three are returned.

## Running locally

```
./runner-scripts/e2e-test/run.sh
```

Requires: `nix` with flakes enabled.  Models are cached after the first run.
The `red_sunset.png` test image is generated at runtime via PIL inside the
indexer nix shell; no binary asset is committed.
